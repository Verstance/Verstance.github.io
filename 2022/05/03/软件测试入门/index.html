1 基础概念

1.1 软件开发

软件的概念：是计算机系统中与硬件相互依存的另一部分，包括程序，数据及其相关文档的完整集合。

软件的十大特性

（1）型态特性：无形的、不可见的逻辑实体。

（2）智能特性：复杂的智力产品。

（3）开发特性：一些工具辅助软件开发工作。

（4）质量特性：软件没有完美的，肯定有缺陷。

（5）生产特性：复制十分简单，生产成本低。

（6）管理特性：软件过程中的管理显得更为重要，也更独特。

（7）环境特性：对计算机系统环境十分依赖。

（8）维护特性：维护体现在升级、优化、功能更新等，甚至全盘重构。

（9）废弃特性：软件并不是由于被“用坏”而被废弃。

（10）应用特性：软件的应用极为广泛。

软件的分类

  • 系统软件（负责管理计算机系统中各个独立的硬件，如服务性程序、语言程序、操作系统、数据库管理系统等）
  • 应用软件（为了某种特定的用途而被开发的软件，如浏览器、小程序等）

软件的生命周期

问题定义→ 可行性研究→需求分析→概要设计→详细设计→编码和单元测试→综合测试→软件维护

软件开发模型

  • 瀑布模型：
    计划→需求分析→设计→编码→测试→运行维护
    特点：
    1.软件界开发的各项活动严格按照线性方式进行；
    2.当前活动接受上一项活动的工作结果；
    3.当前活动的工作结果需要进行验证。
    缺点：
    a.由于开发模型是线性的，增加了开发的风险；
    b.早期的错误可能要等到开发后期的阶段才能发现。

  • 原型模型：
    客户与开发公司紧密联系，开发周期长。开发会受到需求变更的影响。
    特点：
    1.实现客户与系统的交互；
    2.进一步细化待开发软件需求；
    3.开发人员可以确定客户的真正需求是什么。

  • 螺旋模型：
    制定计划→风险分析→实施工程（瀑布模型）→客户评估
    特点：
    1.将瀑布模型与快速原型模型结合起来；
    2.强调了其他模型所忽视的风险分析；
    3.每一次都包括4个步骤：制定计划、风险分析、实施工程、客户评估。
    缺点：
    a.过于强调风险，但要求许多客户接受并相信这种分析，是不容易的。

  • 敏捷模型：
    以人为核心、迭代、循序渐进的开发方法。
    特点：
    1.开发周期短；
    2.增量开发；
    3.由程序员和测试人员编写的自动化测试来监控开发进度；
    4.通过口头沟通、测试和源代码来交流系统的结构和意图；
    5.编写代码之前先写测试代码，也叫做测试先行。
    缺点：
    a.人员素质要求高，团队组件较难；
    b.对测试员要求完全掌握各种脚本语言编程，能执行单元测试、自动化测试。

软件开发文档
需求分析文档：作为依据
概要设计文档：是否覆盖了需求
详细设计文档：方案、策略、架构、体系、接口名称+调用方式、数据库的结构样式
测试设计文档：测试内容，测试的功能点，功能交互，联合测试
测试用例：规范条文
测试报告：测试用例数量，缺陷情况，是否可以上线

1.2 软件测试

软件测试：在规定的条件下对程序进行操作，发现程序错误，衡量软件质量，对其是否能满足设计要求进行评估的过程。

标准定义：软件测试是用人工或自动的手段来运行或测试某个软件系统的过程，其目的在于检验它是否满足规定的需求或弄清预期结果与实际结果之间的差别。

软件测试目的：发现问题，是否满足需求。

冒烟测试：在对一个新版本进行系统大规模地测试之前，先验证软件的基本功能是否实现，是否具备可测性。

α测试：验收测试的一种，指用户、测试、开发等共同参与的内部测试。

β测试：验收测试的一种，指内测后的公测，完全交给最终用户测试。

软件测试模型

V模型：瀑布模型的一种改进，计划，分析，设计，编码，测试和维护。

W模型： v模型的一种并行改进，测试与开发同步进行，尽早发现问题。

H模型：不存在严格的次序关系，各阶段间可以反复触发，迭代、增量。相当于将测试活动完全独立出来，形成一个完全独立的流程，将测试准备活动和测试执行活动清晰地体现出来。只要具备了测试条件，就可以开始测试；尽早准备，尽早执行。

X模型：多层次、合并；熟练性测试，分块化、探索性测试。

软件测试覆盖率
度量测试完整性的一个手段，同时也是测试有效性的一个度量
覆盖率 = （至少被执行一次的item数）/item总数
特点：

  1. 通过覆盖率数据，可以检测我们的测试是否充分
  2. 分析出测试的弱点在哪些方面
  3. 设计能够增加覆盖率的测试用例，有效提高测试质量，测试成本会随覆盖率的增加而增加

需求覆盖 = （被验证到的需求数量） / （总的需求总数）
用例覆盖 = （验证通过的用例数量） / (总的用例总数)

简单的测试覆盖率：本次测试执行的用例数/所有用例数
一般对于大型系统测试要求覆盖率100%
审核：抽样验收

基于产品的测试覆盖率：已测试需求点/设计所有需求数
需求覆盖率达到100%
审核：抽样验收

基于白盒的测试覆盖率：大多工具判断语句覆盖，单元测试代码覆盖代码行/总代码行
更多考察研发人员，要求覆盖率达到80%
缺：覆盖率数据只能代表测试过的代码，不能代表是否测试好这些代码

基于自动化的测试覆盖率：自动化覆盖的测试场景（测试用例）/所有测试场景（用例）
八二 原则，自动化注重20%的核心功能
用途：回归验证，考虑用例设计
最终意义：测试停止标准，减少上线后的风险

测试覆盖率是考验考核的核心标准

软件测试原则

  1. 所有的测试都应追溯到用户需求

  2. 尽早启动测试工作

  3. Pareto(帕累托)法则【28效率法则】应用于软件测试

  4. 穷尽测试是不可能的

  5. 杀虫剂怪事。测试越多，其对测试的免疫力越强的现象

  6. 前进两步，后退一步。缺陷修复总会以（20-50）%几率引入新缺陷

  7. 三心二意。细心、信心、耐心、团队合作的沟通意识、时刻保持怀疑的态度且有缺陷预防意识

软件工程规范

国内通用的软件工程标准主要有ISO9000及CMM。
国内ISO9000：控制和预防思想
CMM：对过程能力的评估结果

测试系统由6个相互关联、相互作用的过程组成。
测试规划、测试设计、测试实施、配置管理、资源管理、测试管理
1.角色的确定
2.进入的准则
3.输入项

2 测试流程

软件测试过程：制定测试计划→设计测试用例→执行测试→提交测试bug→修改追踪 →结束

2.1 测试环境

软件测试环境搭建原则
搭建测试环境前
1.确定测试目的，功能测试、稳定性测试、还是性能测试，测试目的不同，搭建环境时侧重点也不同。
　　功能测试：不需要大量的数据，需要覆盖率高，测试数据要求尽量真实。
　　性能测试：可能需要大量存量数据或者与实际硬件环境尽可能相似的硬件配置。
2.测试软件环境尽可能的模拟真实环境
　　尽可能的模拟用户使用环境，选用合适的操作系统和软件平台；
　　了解符合测试软件运行的最低要求及用户使用的硬件配置；
　　了解用户常用的软件，避免所有的配置所有操作系统下都要进行测试，没有侧重点，浪费时间；
　　产品化的测试则需要考虑兼容性的方案。

3.营造独立的测试环境
　　不同的项目、不同的公司会对测试环境的独立性有不同的要求；
　　测试过程中尽量保证测试环境独立，不会受到其他测试人员以及项目研发人员的影响。
4.构造可复用的测试环境
　　通过数据备份或数据隔离的方式。
　　重复运用一套测试环境进行多版本多时间段的测试。
5.线下构建
　　独立测试服务器或虚拟机
　　测试环境配置
　　测试项目导入
测试环境配置
配置java环境（下载jdk并配置环境变量）
下载并安装中间件（tomcat、jetty或其他）
安装数据库并导入初始化脚本
Docker模式
构建属于自己的image
一键deploy

环境建设思路
1、考虑点：用途、使用成本、维护成本
2、基本架构
包括研发环境测试环境，
研发环境：用于研发自测、集成测试
测试环境：用于日常单系或两辆微服务之间测试，可同时集成自动化测试回归
联测环境：完备环境，用于大型联测，比如淘宝架构迁移，要大型联测
外联环境：稳定版本环境，用于外部商户等联调，比如外部的金融机构与支付宝进行外联所需要的测试环境
灰度/沙箱环境：用于生产数据测试，仿真测试

2.2 测试计划

测试策划过程：

  1. 进行测试需求分析
  2. 确定需要测试的内容或质量特征
  3. 明确测试的充分性要求
  4. 提出测试的基本方法
  5. 确定测试的资源和技术需求
  6. 进行风险分析与评估
  7. 根据上述分析结果制定测试计划
  8. 根据测试计划开展相应的测试控制活动

需求分析

1、测试工程师参与需求分析，对需求了解很深刻，减少与开发人员的交互，节省时间

2、早期确定测试用例的编写思路，为测试打好了基础

3、可以获取一些测试数据，为测试用例设计提供帮助

4、可以发现需求不合理的地方，降低测试成本

需求测试的作用

1、测试需求的分析用来确定整个测试工作，明确测试对象以及测试工作的范围和作用，并作为测试覆盖的基础

2、被确定的测试需求项必须是可核实的，测试需求必须有一个可观察、可评测的结果

3、如果无法核实的需求就不是测试需求，这个需求不用进行测试

4、测试需求分析还包括与客户的交流以澄清某些混淆

5、明确哪些需求更重要

6、确保风险承担者尽早地对项目达成共识

7、并对将来的产品有个清晰的认识

8、测试需求是制定测试计划的基本依据

9、测试需求是设计测试用例的指导

10、确定了解要测什么，测哪些方面才能有效设计用例

需求验证

  1. 审查需求文档
  2. 以需求为依据编写测试用例
  3. 确定合格的标准

测试策略是描述测试项目和测试任务之间的关系：测什么，如何测，如何协调测试资源和测试时间。策略制定的是否合理高效会对测试项目的进度产生很大的影响。

测试策略要素：测试安排发布计划—测试范围—测试资源—测试环境—测试方法—用例设计方法—文档管理—风险管理—上线跟踪验证

1.测试安排发布计划
罗列测试项目本身重要的里程碑
每个里程碑都需要有明确的结束时间
规划优先级（这样可以保证产品总质量过关）

2.测试范围（按优先级排列）
in scope 与out of scope（范围内与范围外）
范围内的需要给定优先级，以对应测试时间不足的情况
范围外的，不需要测试的模块，需要给出原因

3.测试资源
测试策略中重要的一环，分为人力与工具两部分

4.测试环境
主要包括推荐环境解决方案，操作系统要求，软硬件要求

5.测试方法
主要为了说明针对测试项目，需要开展哪些类型的测试
例如功能测试是必须的，非功能测试是可选的

6.文档管理
一个完整的产品，文档是很重要的一环
包含安装，升级，用户指南；
需要经过完整的测试才能给客户
差的文档可能会误导用户，this is error

7.风险管理
风险管理模块需要列出已知的可能会出现不确定性的因素

2.3 测试评审

前提：仅从文档、沟通获取信息，可能造成信息不对称，认识片面，理解错误或不深入等问题；缺少同行交叉评审和开发评审机制，无法充分发挥集体智慧，个人的思维难以突破，可能会出现测试遗漏的问题。

评审目的：

  • 呈现测试的工作
  • 与开发达成共识
  • 不同的思维方式碰撞出火花，借鉴别人的思考方式培养这样的行为模式：愿意为团队或他人出谋划策，发挥团队协作，最大限度发挥个人长度，取长补短。

评审重点：

  • 采用的测试方法
  • 等价类划分的依据
  • 测试数据的选取和准备方法
  • 流程测试的路径组合
  • 数据比对选取的对象和数据检查点
  • 是否需要模拟数据及模拟数据的方法
  • 基于风险的测试取舍

3 测试用例

3.1 用例设计

测试设计：将概括的测试目标转化为具体的测试条件和测试用例的一系列活动

测试分析和设计的主要任务：

  1. 评审测试依据（需求，系统架构，设计和接口说明）
  2. 评估测试依据和测试对象的可靠性
  3. 通过对测试项、规格说明、测试对象行为和结构的分析，识别测试条件并确定优先级
  4. 设计测试用例，并确定优先级
  5. 确定测试条件和测试用例所需的必要的测试数据

确定测试条件：

  1. 依据在测试策略或测试计划中确定的测试技术（如自动化、性能测试等）

  2. 通过对测试依据和测试目标的分析，可以确定需要测试的内容，获得测试条件

    需求→测试内容

测试用例：指引测试的步骤文档。应具备可重复性、可验证性、需求可追踪性。
包括以下关键点

  1. 前提条件，如项目或局部测试环境的需求，及其交付计划
  2. 测试步骤
  3. 测试数据
  4. 预期结果

测试用例常用设计方法

  • 等价类划分法
  • 边界值法
  • 因果图设计法
  • 判定表设计法
  • 正交实验法
3.1.1 等价类划分法与边界值法

等价类划分法：把程序的输入域划分成若干部分，然后从每个部分中选取少数代表性数据当作测试用例，每一类的代表性数据在测试中的作用等价于这一类中的其他值。

等价类划分原则：
。如果输入条件规定了取值的范围或值得个数，则可确定一个有效等价类和两个无效等价类
。如果一个输入条件说明了一个“必须成立”的情况，则可划分一个有效等价类和一个无效等价类
。如果输入条件规定了属入数据的一组可能的值，而且程序是用不用的方式处理每一种值，则可为每一种值划分一个有效等价类，并划分一个无效等价类。
。如果我们确定，已划分的某等价类中分各个元素（例子）在程序中的处理方式是不同的，则应据此将等价类进一步划分成更小的等价类。
。在确立了等价类之后，建立等价类表，列出所有划分出的等价类

两位数加法器等价类划分实战

  1. 根据需求分为三个等价类。一个有效等价类-99到99，两个无效等价类<-99 >99

  2. 建立等价类表 包含 功能项 有效等价类 编号 无效等价类 编号

  3. 设计用例，使它只覆盖尽可能多的有效等价类和一个无效等价类，重复这一步骤，从而使无效等价类均被测试用例覆盖

  4. 细化等价类 如果发现对于不同的数可能有不同的处理模式，需要再进行细化

等价类的特点
。测试相同的内容
。如果等价类中的一个测试能够捕获一个缺陷，那么选择该等价类中的其他测试也能捕获缺陷；如果等价类中的一个测试不能捕获缺陷，那么选择该等价类中的其他测试也不会捕获缺陷
。如果正确地划分等价类，可以大大降低测试用例的数量，测试会准确有效
。如果错误地将两个不同的等价类当做一个等价类，那就回遗漏一种测试情况；相反的，把同一个等价类看做了两个不同的等价类，那么测试就会是冗余的

等价类划分要注意的问题

  1. 不到那要考虑有效等价类，也要考虑无效等价类
  2. 仔细划分，审查划分
  3. 过于粗略可能会漏掉软件缺陷
  4. 要组织评审

按照等价类划分设计测试用例

边界值分析法：一种补充等价类划分的测试用例设计技术，它不是选择等价类的任意元素，而是选择等价类边界的测试用例。

边界值分析法不仅重视输入条件边界，而且也从输出域导出测试用例

边界值设计的原则：如果输入条件规定了取值范围，应以该范围的边界内及刚刚超过范围的边界外的值作为测试用例，如以a和b为边界，测试用例应当包含a和b及略大于a和略小于b的值。

同样的测试需求：余额宝提现快速到账问题

3.1.2 因果图法与判定表法

因果图→判定表

因果图方法就是从程序规格说明书的描述中找出因（输入条件）和果（输出结果或程序状态的改变）。

判定表（Decision Table）是分析和表达多逻辑条件下执行不同操作的工具。在程序设计发展初，判定表就已被当做编写程序的辅助工具。它可以把复杂的逻辑关系和多种条件组合的情况表达的既具体又明确。

将因果图转换为判定表，为决策表中的每一列设计一个测试用例
这种方法考虑了输入情况的各种组合以及各个输入情况之间的相互制约关系。

判定表

判定表通常由四个部分组成：

  • 条件桩（Condition Stub）：列出了问题的所有条件，通常认为列出的条件的次序无关紧要
  • 动作桩（Action Stub）：列出了问题规定可能采取的操作，这些操作的排列顺序没有约束
  • 条件项（C Entry）：列出针对它所列条件的取值，在所有可能情况下的真假值
  • 动作项（A E）：列出在条件项的各种取值情况下应该采取的动作

设计步骤：

  1. 分析软件规格说明中那些是原因（输入或它的等价类），哪些是结果（输出），并给每个原因和结果赋予一个标识符；

  2. 分析软件规格说明中语义的内容，找出原因与结果之间、原因与原因之间对应的关系，根据这些关系话出因果图；

  3. 有些组合情况不可能出现，为表明这些情况，在因果图上使用一些记号表明约束或限制条件；

  4. 把因果图转换为判定表；

  5. 根据判定表的每列设计用例。

3.1.3 正交实验法

正交实验法是从大量的试验点中挑选出适量的、有代表性的点，应用依据伽罗卡瓦理论推导出的“正交表”，合理的安排实验的一种科学的试验设计方法。

指标：判断实验结果优劣的标准
因子（因素Factor）：所有影响试验指标的条件
因子的状态（水平Level）：影响试验因子的，叫做因子的状态（因子变量的取值）

正交实验法设计步骤

  1. 提取功能说明，构造因子——状态表
  2. 加权筛选，生成因素分析表
  3. 利用正交表构造测试数据

3.1.4 场景法

场景法原理：

现在的软件几乎都是用事件触发来控制流程的，事件触发时的情景就形成了场景，而同一事件不同的触发顺序和处理结果就形成了事件流。

在软件设计方面的思想可以引入到软件测试中，可以生动地描绘出事件触发时的情景，有利于设计测试用例，同时使测试用例更容易理解和执行。

场景法基础设计

经过用例的每条路径都用基本流和备选流来表示，直黑线表示基本流，是经过用例的最简单的路径。

多角度切入系统，将系统划分成一块一块的进行测试， 确保整个大的测试完整性

3.2 测试执行

3.2.1 执行过程

测试执行阶段的主要任务：

  1. 确定测试用例的优先级
  2. 开发测试规程并确定优先级，创建测试数据，同时也可以准备测试用具和设计自动化测试脚本
  3. 根据测试规程创建测试套件，以提高测试执行的效率
  4. 确认已经正确搭建了测试环境
  5. 根据计划的执行顺序，通过手工或使用测试工具来执行测试过程
  6. 记录测试执行的结果，以及被测软件、测试工具和测试件的标识和版本
  7. 将实际结果和预期结果进行比较
  8. 对实际结果和预期结果之间的差异，作为事件上报，并且进行分析以确定引起的差异的原因
  9. 缺陷修正后，重新进行测试活动

测试准入标准

  1. 开发编码结束，并在开发环境已完成单元测试
  2. 需求上规定的功能均已实现；如没完全实现，请提供测试范围
  3. 已完成集成测试，被测系统的基本流程可以走通，界面上的功能均已实现，经过代码评审并符合软件编码规范
  4. 开发提交最新版本代码，以此为基线，提交并通知测试组进行测试
  5. 兼容性测试要求明确
  6. 安全测试和性能测试范围和要求

测试暂停、停止

  1. 先进行冒烟测试，若发现重大缺陷或bug过多时、或者流程卡壳导致基本流程无法走通，测试无法正常进行，可以暂停测试并返回开发
  2. 被测项目需要调整而暂停的，测试也相应暂停
  3. 存在其他优先级更高的任务时，可以向领导申请暂停测试
  4. 被测系统经过系统测试，到达系统准出标准，可以停止测试

3.2.2 缺陷管理

软件缺陷

  1. 软件未达到产品说明表明的功能
  2. 软件出现了产品说明指明不会出现的错误
  3. 软件功能超出产品说明指明范围
  4. 软件未达到产品说明虽未指出但应达到的目标
  5. 认为软件难以理解、不易使用、运行缓慢，或最终用户认为不好

缺陷产生的原因：沟通交流不够，需求不断变化，软件的复杂性，程序设计错误，文档不完善，工期短，任务大，软硬件支持不完善

发现缺陷（后果）

  1. 用户体验不好
  2. 界面上有明显错误信息
  3. 功能不完备，没有按照需求说明编码，致使功能缺失
  4. 功能不完善，不能正常运行或者运行中出现崩溃
  5. 逻辑不正确，与需求、用例不符
  6. 模块之间的交互性不好
  7. 程序性能不好，不能负载

缺陷报告注意事项

缺陷报告：

  1. 缺陷报告是对缺陷进行记录、分类和跟踪的文档
  2. 软件测试人员的任务之一就是书写良好的软件缺陷报告
  3. 提供准确、完整、简洁、一致的缺陷报告是体现软件测试的专业性、高质量的主要评价指标
  4. 通常，缺陷报告的直接读者是软件开发人员和质量管理人员，除此之外，来自市场和技术支持等部门的人也可能需要查看缺陷情况

缺陷报告之无法重现的BUG

  • 首先，应当对这样的缺陷进行详细的记录，并尽快提交开发人员
  • 其次，对于寻找难以再现的缺陷要合理地安排时间，要考虑到测试项目的整体进度，对一时难以再现的缺陷可以暂时搁置，以保证项目的正常进度
  • 最后在测试过程中对未再现缺陷予以关注

缺陷报告的写作准则（5C）

  1. Correct(准确)：每个组成部分的描述准确，不会引起误解
  2. Clear(清晰)：每个组成部分的描述清晰，易于理解
  3. Concise(简洁)：只包含必不可少的信息，不包括任务多余的内容
  4. Complete(完整)：只包含复现改缺陷的完整的步骤和其他本质信息
  5. Consisten(一致)：按照一致的格式书写全部缺陷报告

缺陷报告的组织架构：

  1. 缺陷的标题，比如容易被搜索到
  2. 缺陷的基本信息，比如我是在哪个项目里面，在哪个版本里，使用什么浏览器等
  3. 测试的软件和硬件环境
  4. 测试的软件版本
  5. 缺陷的类型
  6. 缺陷的严重程度
  7. 缺陷的处理优先级
  8. 复现缺陷的操作步骤
  9. 执行某个流程中的实际结果的描述
  10. 期望的正确结果描述
  11. 注释文字和截取的缺陷图像，更甚至是录制视频

缺陷标题

  1. 尽量按缺陷发生的原因与结果方式书写（“执行完A后，发生B”或者“发生B，当A执行完后”）
  2. 避免使用模糊不清的词语，例如“功能终端”“功能不正确”“行为不起作用”等。应该使用具体文字说明功能如何中断，如何不正确，或如何不起作用
  3. 为了方便搜索和查询，请使用关键字
  4. 为了便于他人理解，避免使用术语、俚语或过分具体的测试细节

复现步骤：包含如何使别人能够很容易的复现改缺陷的完整步骤。为了达到这个要求，复现步骤的信息必须完整的、准确的、简明的、可复现的

  1. 提供测试的预备步骤和信息
  2. 简单地一步一步引导复现该缺陷
  3. 每一个步骤尽量只记录一个操作
  4. 每个步骤前使用数字对应步骤编号
  5. 尽量使用短语和短句，便面复杂句型和句式
  6. 复现的操作步骤要完整、准确、简短
  7. 没有缺漏任何操作步骤
  8. 每个步骤都是准确无误的
  9. 没有任何多余的步骤
  10. 将常见步骤合并为较少步骤
  11. 只记录各个步骤是什么，不需要包括每个步骤的执行结果

缺陷报告的原则

3.2.3 易用性测试

易用性测试：

  1. 易用性测试是指用户使用软件时是否感觉方便，比如是否最多点击鼠标三次就可以达到用户的目的
  2. 易用性和可用性存在一定的区别，可用性是指是否可以使用，而易用性是指是否方便使用
  3. 易用性测试包括针对应用程序的测试，同时还包括对用户手册系统文档的测试。通常采用质量外部模型来评价易用性

易用性测试内容：易理解性、易学习性、易操作性、吸引性、依从性

3.2.4 兼容性测试

兼容性测试：

  1. 兼容性测试简称CTS，指对所设计程序与硬件、软件之间的兼容性的测试
  2. 一般来说，兼容性指能同时容纳多个方面，在计算机术语上兼容是指几个硬件之间、几个软件之间或是软硬件之间的互相配合程度
  3. 对于我们测试来说，通俗一点的理解可以认为是被测软件在不同的硬件平台（PC||Mobile）,不同的软件（浏览器），不同的操作系统平台、不同的网络环境是否能够很友好运行的测试

WEB兼容性测试方向： 浏览器兼容性 、操作系统兼容性
测试方法：

  • 人工测试：测试工程师测试主流浏览器和常用操作系统测试主流程和主界面，看看主流程和主界面是否有问题
  • 第三方测试工具：部分情况下，部分浏览器可以依赖第三方工具辅助测试

浏览器根据内核不同，测试选型

  • Chrome：Webkit内核*1 & Blink内核*1
  • FireFox：最新版本（静默更新）
  • IE：7-11
  • Safari：mac版本单独测试
  • Edge：window10
  • 360 安全浏览器：双核版
  • 搜狗等其他浏览器任选一
  • 如有需要Linux系统下FireFox、ChromeOS下chrome

三方工具：IETESTER，BrowserShots（只能在线），Super Preview

APP兼容性测试方向：硬件设备兼容性、操作系统版本兼容性
测试方法：

  • 人工测试：测试工程师测试主流手机设备对设备主流程和主功能进行验证测试

4.1 白盒测试

4.1.1 代码审查

范围：1.业务逻辑的审查 2.算法的效率 3.代码风格 4.编程规则

方法：互查（互相检查对方的代码）、走查（从头到尾将写好的程序检查一遍）、代码审查（通过会议方式对代码进行校验）

规则：

  1. Java最基本语句的使用
  2. 重载函数的审查
  3. 内存分配和管理
  4. 程序性能的审查
    • 减少创建对象
    • 减少循环体的执行代码
    • 提高处理异常出错的效率
    • 减少I/O操作时间
4.1.2 逻辑覆盖

单元测试用例设计→白盒测试

  1. 对程序模块所有独立的执行路径至少要测试一次
  2. 对所有逻辑判定，其结果为真、假两种情况至少要测试一次
  3. 对程序进行边界检查（常见的如数据越界检验）
  4. 检验内部数据结构的有效性

白盒测试最主要测试方法：逻辑覆盖法

逻辑覆盖分类：

  • 语句覆盖：设计若干测试用例，运行被测程序，使程序中每个可执行语句至少被执行一次
    不足：不能准确判断运算中的逻辑关系错误
  • 判定覆盖：使每个判断的取真值和取假值都至少经历一次
    不足：条件中有or条件的容易被忽略
  • 条件覆盖：使每个条件的取真值和取假值至少经历一次
    不足：会遗漏程序逻辑路径或判定错误，依旧存在较大风险
  • 判定-条件覆盖：确保设计足够或精巧测试用例，可使得判断条件中所有条件可能值至少被执行一次，同时，所有判断可能结果也至少执行一次
  • 条件组合覆盖：让每个条件出现真假的所有可能组合都至少出现一次

6.5 项目提测

冒烟测试

冒烟测试（预测试）：确认模块是否符合基本逻辑满足基本功能。

适用于：第一次提测，一轮修改涉及更改量大，需求变更后。

冒烟测试不超过总时长的1/10

冒烟测试分类：

  1. 核心流程冒烟：要求100%通过
  2. 主流程冒烟（涉及的功能点）：不能超过30%的场景未实现
  3. 探索式冒烟：半小时随机测试，发现bug超过10个打回

不符合冒烟通过，视为不进入测试阶段，测试打回，需要重新提测，重新冒烟

禅道中的套件很好地在冒烟测试阶段对需要进行核心流程测试用例进行管理，

项目 版本 套件 测试单 （根据套件进行关联测试用例）

7 测试总结

测试报告（测试阶段最后文档产出物）

  • 包含测试过程和结果，产品质量和测试过程的评价
  • 基于测试中的数据采集，对最终的测试结果 / 发现的问题和缺陷进行分析（为纠正问题提供依据）
  • 为软件验收测试和最终交付打下基础

优秀的测试人员应该具备良好的文档编写能力！

基本内容：

  • 报告信息：类似报告文档的抬头，可算作标题 / 备注，测试总体的高度概括

  • 引言：编写目的（预期的读者范围）、项目背景（简要说明项目目标）、系统简介（设计说明书 / 需求文档，框图和网络拓扑图辅助说明）、评测产品（描述测试对象，包括文件 / 程序所在SVN / Git 路径、版本号等信息）

  • 测试概要：测试情况简介（包含测试声明、测试范围、测试目的等信息），比如测试用例设计方法、测试环境与配置、应用的测试方法和工具

  • 测试结果与分析：主要汇总各种数据进行度量（包括对测试过程和软件产品的质量度量和评估），测试执行情况与记录，测试用例执行情况（执行X条 / 通过Y条 / …）、测试组织（可列出简单的测试组架构图，包括测试组架构（分组、用户参与等情况）、测试经理、主要测试人员、参与测试人员）、覆盖分析（需求覆盖率 = 经过测试的需求 / 需求规格说明书中所有需求，需求覆盖率通常要达到100%；测试覆盖率 = 执行用例个数 / 需求用例个数，分析未执行 / 延后执行原因，大型项目需要将手工测试与自动化测试相结合，测试覆盖率至少要达到90%以上 ）、缺陷分析（用例质量 = 缺陷总数 / 测试用例总数 × 100%，缺陷密度 = 缺陷总数 / 功能点总数，概括测试缺陷的等级并针对其进行分析，测试曲线图描述的缺陷走势应该越来越低，趋向收敛）、残留缺陷和未解决问题（BUG编号、缺陷概要、原因分析（如何引起、后缺陷果、未解决原因）、预防和改进措施）

  • 测试结论与建议：测试执行是否充分（可增加对安全性、可靠性、可维护性和性能描述）、是否可以以进入下一阶段项目 / 迭代、对测试风险的控制措施和成效、测试目标是否完成、测试是否通过；建议（说明测试揭露的软件缺陷以及可能带来的影响、可能存在的潜在缺陷以及后续工作，对整个流程（缺陷修改、产品设计、沟通方式）中改进方面的建议）

  • 测试限制

验收测试策略

  1. Alpha测试：由用户在开发环境下进行的测试，在系统开发接近完成时进行的测试
  2. Beta测试：由软件的多个用户在一个或多个用户的实际使用环境下进行的测试，是一种模拟真实的使用环境从而发现缺陷的一种测试
